# Plan Project Refactoring

Keep the entire workflow read-only. Do not edit, generate, move, or delete repository files; invoke write-capable Unity commands; install or refresh packages or skills; change Jira, Git refs, branches, worktrees, or index state; create repositories; publish; or deploy. A proposal never authorizes implementation.

1. Resolve the exact repository or worktree requested by the user. Capture `git status --short --untracked-files=all` as the byte-for-byte baseline and preserve every pre-existing change.
2. Read applicable `AGENTS.md`, `CLAUDE.md`, primary project router, directory-scoped instructions, factual architecture, package workflow, validation, and safety guidance. Keep the user's requested analysis scope explicit.
3. Before source inventory, run `scripts/ai_refactor_jira_cli.py overlap --format json` from this installed skill directory. Require `scope=project`, exact `states=[todo, progress, done]`, and `complete=true`, then run `scripts/ai_refactor_jira_cli.py detail <ISSUE-KEY> --format json` for every returned key. This is read-only evidence across all assignees. It intentionally excludes statuses outside the configured mappings, including QA-board or future states. Missing config, credentials, permission, status mappings, tools, terminal pagination proof, or any issue detail stops the entire workflow before inventory or proposal output. Never fall back to a current-user list or claim no overlap from partial evidence.
4. Resolve AI Code Convention from `Packages/com.actionfit.ai-codeconvention`; otherwise use exactly one `Library/PackageCache/com.actionfit.ai-codeconvention@*` without editing it. Read its `AI_GUIDE.md`, `references/unity-code-authoring-rules.md`, exact `AI Code Convention profile: <id>` selector, selected profile reference, and `references/owner-routing.md`. Use `portable-core` only when the selector is absent; report duplicate or unknown selectors instead of guessing. The convention owns `AFCC-PCR-001`; a product marker never selects the profile.
5. Resolve every installed API-owner guide required by observed concrete APIs. The effective AI Code Convention and installed owner guide own rule meanings and factual API surfaces; this skill never invents an API or broadens `AFCC-*` rules.
6. Read `references/refactor-analysis-contract.md`. Use its statuses, Jira overlap categories, blocking rules, evidence requirements, target-tree notation, package candidate fields, output sections, and confidence rules.
7. Run `scripts/refactor_inventory.py --root <repository-root> --format json` from this installed skill. Treat every result as a discovery candidate. Do not redirect output into the repository or use a tool that writes caches there.
8. Read `productComposition` from the inventory. `absent` keeps the generic target. For `valid`, directly verify the exact complete trimmed `AI Product Composition Root: <package-id>` and `AI Refactor target: package-oriented-product` lines inside the root guide's `## Package Identity`, the sibling manifest name, embedded-before-PackageCache resolution, uniqueness, and the separately selected profile before applying `AFCC-PCR-001`. For `invalid`, report its diagnostics as missing evidence and do not select or infer a root.
9. Inspect each material candidate directly before making a claim. Cite repository-relative `path:line`, identify the observed owner, state and lifetime, dependency direction, consumers, and concrete project or SDK coupling. Use `Violation` only when direct evidence conflicts with one cited effective `AFCC-*` rule. Otherwise use `Observed`, `Candidate`, `Missing evidence`, or `Deferred`.
10. In `Product composition and project shell`, keep package manifest, asmdef, and directly evidenced runtime ownership graphs separate. Classify supported nodes as `Composition`, `Product`, `Reusable`, `Project Shell`, or `Exception`; report cycles, unresolved edges, package-to-project reverse dependencies, and unproved `Assets` residue without classifying folders or inventory hits automatically.
11. Compare every Jira title and description with directly evidenced package IDs, paths, symbols, owners, dependency directions, and extraction boundaries. Classify each material match as `Exact overlap`, `Partial overlap`, or `Related`; use `No overlap` only after every returned issue was inspected. An `Exact overlap` or material `Partial overlap`, including configured `done` work, stops before target DAG, package candidates, or ordered phases and asks whether to reuse the issue, exclude the overlap, or plan only a proven residual gap. A `Related` result records the boundary and may continue. Do not use an automatic numeric score as the verdict.
12. Build the current ownership/dependency graph and an `AFCC-TRE-001` target tree-oriented DAG. Identify composition roots, coherent feature and service nodes, shared nodes with multiple consumers, dependency edges, cycles, and project-only bindings. Do not equate the target with a Unity Transform tree, inheritance tree, folder tree, single root, or one instance per displayed node.
13. Evaluate package candidates through `AFCC-PKG-001`: project-neutral rules/state/lifecycle, one-way dependencies, excluded project types/scenes/assets/save keys/SDKs, reuse evidence, and whether engine/UI/adapter separation is genuinely warranted. Package count is not a goal.
14. Evaluate ports and adapters through `AFCC-PRT-001` and `AFCC-INT-001`. Require a real external production capability, keep the port consumer-oriented and narrow, bind project adapters at a composition root, and reject speculative interfaces, test-only seams, DI containers, service locators, and unsafe no-op defaults.
15. Return these sections in order only when no blocking overlap exists: `Scope and convention`, `Jira overlap check`, `Product composition and project shell`, `Current findings`, `Current ownership graph`, `Target tree-oriented DAG`, `Package candidates`, `Ports and project adapters`, `Ordered phases`, `Migration and compatibility risks`, `Validation plan`, `Confidence and missing evidence`, and `Deferred or out of scope`. Every material finding and proposal maps to evidence and applicable `AFCC-*` IDs.
16. Make phases independently reviewable and reversible. Include compatibility seams, serialized/public API/save/asset implications, negative tests, rollback points, and what must be separately authorized. Never describe source movement, asset migration, publishing, or project-wide replacement as already approved.
17. Capture the same Git status command after analysis and compare it byte-for-byte with the baseline. If durable repository state changed, identify the paths, mark the no-write contract failed, and stop without presenting the proposal as complete.

Report inventory/tool limitations and unverified runtime behavior. Do not claim full source compliance, successful compilation, gameplay correctness, or package readiness without separate evidence.
